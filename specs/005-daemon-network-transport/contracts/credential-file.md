# Contract: Credential File Schema

## Overview

A credential file is a JSON document generated by `sigilctl credential add <name>` on the daemon machine. It is transferred to the host machine (out-of-band: scp, shared clipboard, mounted folder) and stored at `$XDG_CONFIG_HOME/sigil-shell/daemon-credential.json` with file permissions `0600`.

The file is **secret**. It contains the plaintext bearer token and must be treated like an API key or private key.

## Schema

```json
{
  "id": "macbook-pro",
  "token": "sghl_a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2",
  "server_addr": "192.168.64.3:7773",
  "server_cert_spki": "sha256/nRWr6HGW8I23TWF9sKxnrXSRvF2WqcssmzXw5vqHFQo=",
  "generated_at": "2026-03-16T14:23:45Z"
}
```

## Field Definitions

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | string | yes | Credential name. Must be non-empty, alphanumeric + hyphens, max 64 chars. Matches the ID on the daemon side. |
| token | string | yes | Bearer token. Format: `sghl_` prefix followed by exactly 40 lowercase hexadecimal characters. Total length: 45 chars. |
| server_addr | string | yes | Default connection target in `host:port` format. May be overridden by sigil-shell settings. |
| server_cert_spki | string | yes | TLS server certificate SPKI fingerprint. Format: `sha256/` followed by base64-encoded SHA-256 hash of the DER-encoded SubjectPublicKeyInfo from the server's TLS certificate. |
| generated_at | string | yes | RFC3339 UTC timestamp of credential creation. Informational only. |

## Validation Rules

- `token`: must match regex `^sghl_[0-9a-f]{40}$`
- `server_addr`: must match `host:port` where port is 1–65535
- `server_cert_spki`: must match `^sha256/[A-Za-z0-9+/]{43}=$` (base64, 32 bytes)
- `generated_at`: must be a valid RFC3339 timestamp

## Token Format Rationale

The `sghl_` prefix (Sigil) allows:
- Secret scanning tools (e.g. GitHub push protection, gitleaks) to detect accidental commits
- Log scrubbers to redact tokens by pattern
- Human recognition in config files and shell history

## SPKI Fingerprint Computation

The `server_cert_spki` field is computed as:

```
fingerprint = "sha256/" + base64(sha256(cert.RawSubjectPublicKeyInfo))
```

Where `cert.RawSubjectPublicKeyInfo` is the DER-encoded SubjectPublicKeyInfo extension from the X.509 certificate (not the full certificate DER). This is the same format used by HTTP Public Key Pinning (HPKP) and is stable across certificate renewals that reuse the same key pair.

**Note**: If the daemon regenerates its TLS key pair (e.g., cert expiry), the SPKI fingerprint changes and all credential files must be regenerated.

## Lifecycle

1. **Created**: by `sigilctl credential add <name>` → printed to stdout → saved by engineer to host
2. **Active**: accepted by daemon on TCP connections
3. **Revoked**: rejected by daemon; credential file on host should be deleted
4. **Expired (implicit)**: if server cert is regenerated, the `server_cert_spki` in old credential files no longer matches — connection rejected at TLS fingerprint check

## Example: Multiple Credential Files

An engineer with multiple machines may have:
```
~/.config/sigil-shell/
├── daemon-credential.json          # active
├── daemon-credential-home.json     # another daemon instance
└── daemon-settings.json            # selects which credential to use
```

The `tcp_credential_path` field in `daemon-settings.json` points to the active credential file.
